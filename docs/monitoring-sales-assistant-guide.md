# 监护销售助理 — Copilot Studio 配置与演示指南

> **目标**：把 `NMI Monitoring Sales Assistant` 配置成可演示的销售支持 Agent，帮助销售团队完成客户拜访准备、方案推荐、异议处理和跟进材料生成。
> **适用对象**：监护产品销售、售前、渠道、客户成功、解决方案架构师。

---

## 1. Agent 定位

监护销售助理不是面向患者的医疗问诊机器人，而是面向销售团队的工作助手。它应该帮助销售人员：

- 快速梳理医院客户画像和采购阶段。
- 把 ICU、急诊、病区、手术室、新生儿等场景转成监护解决方案方向。
- 针对院长、设备科、信息科、护理部、科室主任生成不同话术。
- 把会议记录整理成 CRM 摘要、下一步行动和风险提示。
- 对外发话术保持合规，不编造型号、价格、注册证、临床数据或竞品结论。

---

## 2. Copilot Studio 创建步骤

1. 在 Copilot Studio 新建 Agent，名称建议为 `NMI Monitoring Sales Assistant`。
2. 将 [agents/monitoring-sales-system-prompt.md](../agents/monitoring-sales-system-prompt.md) 中的 `System Prompt` 代码块粘贴到 Instructions。
3. 打开 Generative Orchestration，让 Agent 能根据用户输入自动选择 Topic。
4. 配置 Welcome Message，使用 prompt 文件中的启动消息。
5. 创建 5 个 Topic：
   - `Sales Discovery`
   - `Solution Recommendation`
   - `Stakeholder Pitch`
   - `Objection Handling`
   - `Follow-up`
6. 如果作为 NMI 主 Agent 的子智能体使用，在主 `NMI Orchestrator` 中添加 Agent-to-agent 委派，并使用本文第 4 节的调用契约。

---

## 3. 推荐知识库

Demo 阶段可以先不接真实产品目录，让 Agent 使用通用产品类别。生产或客户演示前，建议接入以下只读知识库：

| 知识库 | 建议来源 | 用途 |
|---|---|---|
| 产品目录 | 已批准产品手册、规格表、注册文件摘要 | 型号、模块、适用场景、接口能力 |
| 销售 Playbook | 内部销售方法论、异议处理模板 | 采购阶段、角色话术、推进动作 |
| 客户案例 | 已批准可外发案例和白皮书 | 价值佐证，避免编造案例 |
| 合规资料 | 法务/合规审批后的声明清单 | 限制绝对化、疗效承诺、竞品贬损 |

建议 Azure AI Search 索引：

| 索引名 | 关键字段 |
|---|---|
| `monitoring-product-catalog` | product_name, category, department, approved_claims, limitations, interface_capabilities |
| `monitoring-sales-playbook` | buying_stage, stakeholder, pain_point, discovery_questions, recommended_assets |
| `monitoring-approved-cases` | case_name, hospital_type, department, approved_summary, external_use_allowed |

## 4. Orchestrator 委派契约

当用户在主 NMI Agent 中提出监护销售相关需求时，Orchestrator 应委派给 Monitoring Sales Agent。

示例结构化参数：

```json
{
  "account_name": "某三甲医院",
  "hospital_tier": "三甲",
  "department": "ICU",
  "bed_count": 40,
  "current_solution": "现有监护设备使用超过 8 年，报警多，中央站扩展困难",
  "buying_stage": "初访",
  "stakeholders": ["ICU 主任", "护理部", "信息科", "设备科"],
  "budget_or_timeline": "下季度立项，年底前采购",
  "competitors": ["已有竞品在接触"],
  "requested_output": "拜访话术和方案方向"
}
```

预期返回：客户画像、推荐方向、角色化价值主张、异议回应、下一步行动、合规提醒。

## 5. 8 分钟演示脚本

### 场景 A：初访准备（2 分钟）

输入：

```text
我要拜访一家三甲医院 ICU，40 张床，现有监护设备老旧。主任抱怨报警太多，信息科担心 EMR 对接复杂。帮我准备初访问题和拜访话术。
```

讲解重点：

- Agent 先识别场景：三甲 ICU 存量替换 + 中央监护升级。
- 输出按角色拆分：ICU 主任、护理部、信息科、设备科。
- 不是直接硬卖产品，而是先收集影响方案的关键问题。

### 场景 B：方案推荐（2 分钟）

输入：

```text
客户计划 ICU 40 床分两期升级，第一期 20 床，要求中央站、报警管理、趋势回顾，后续可能接 EMR。请给出推荐方向。
```

讲解重点：

- 推荐“模块化床旁监护 + 中央监护 + 分阶段接口评估”。
- 把一期、二期、接口、培训和维保拆开，便于客户立项。
- 合规提醒会标出：具体型号、接口承诺、价格需查正式资料。

### 场景 C：异议处理（2 分钟）

输入：

```text
客户说竞品价格更低，而且他们已经用惯了。帮我准备合规回应，不要贬低竞品。
```

讲解重点：

- Agent 不攻击竞品，转向总拥有成本、上线风险、培训成本、服务响应、扩展能力。
- 输出可直接用于销售内部准备，但提醒外发前走合规审批。

### 场景 D：会后跟进（2 分钟）

输入：

```text
根据这次会议，生成一封会后跟进邮件和 CRM 摘要：客户关注报警管理、中央站扩展、EMR 对接；设备科要产品资料；信息科要接口清单；下周安排现场勘查。
```

讲解重点：

- 同时生成外发邮件和内部 CRM 摘要。
- 下一步动作明确到负责人和时间。
- 自动提醒删除未授权价格、竞品和联系人敏感信息。

## 6. 期望输出示例

用户输入：

```text
三甲医院 ICU 40 张床，主任关心报警疲劳，护理部关心交接班效率，信息科关心 EMR 接口，设备科关心预算。请给出拜访话术。
```

Agent 应输出类似：

```markdown
## 客户画像
三甲医院 ICU 存量监护升级机会，核心矛盾是临床报警管理、护理效率、系统互联和预算可控之间的平衡。

## 推荐方向
| 模块 | 建议 | 适用理由 | 待确认 |
|---|---|---|---|
| 床旁监护 | 模块化床旁监护配置，按 ICU 高 acuity 场景设计 | 便于覆盖多参数监测和后续扩展 | 现有参数模块、床旁网络、备用机比例 |
| 中央监护 | 中央站 + 报警分级 + 趋势回顾 | 支撑护士站集中查看和交接班回顾 | 护士站布局、床位分组、报警策略 |
| IT 集成 | 先做 EMR 接口评估，再分阶段对接 | 降低一次性上线风险 | EMR 厂商、接口标准、网络安全要求 |

## 角色化价值主张
| 角色 | 关注点 | 建议话术 |
|---|---|---|
| ICU 主任 | 报警疲劳、重症安全 | 我们建议先梳理报警分级和参数策略，再评估中央监护能力，目标是让关键报警更可见、趋势回顾更方便。 |
| 护理部 | 交接班效率 | 可以重点演示趋势回顾、床位集中查看和事件记录，帮助护理团队减少重复查找信息。 |
| 信息科 | EMR 对接、网络安全 | 我们先提供接口能力清单并安排技术评估，任何接口承诺以正式技术文件和现场评估为准。 |
| 设备科 | 预算、维保、可扩展 | 可按一期核心 ICU、二期扩展床位的方式规划，便于控制初期投入并保留扩展空间。 |
```

## 7. 上线前检查

- [ ] 是否接入了正式、已批准的产品目录。
- [ ] 是否禁止模型编造型号、价格、临床数据、客户案例。
- [ ] 是否配置了销售文案合规审查路径。
- [ ] 是否明确外发邮件、投标材料、价格承诺必须走审批。
- [ ] 是否配置了敏感信息脱敏提示。
- [ ] 是否限制 CRM 写入权限，避免 Agent 自动覆盖关键商机字段。
